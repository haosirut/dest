# Архитектура VaultKeeper P2P

## Обзор системы P2P-хранилища

**VaultKeeper P2P** — это децентрализованная одноранговая система хранения данных, спроектированная для обеспечения отказоустойчивого, приватного и экономически эффективного хранения файлов без единого центра отказа. Система построена на базе протокола libp2p и использует распределённый реестр (ledger) с Merkle-деревьями для обеспечения целостности данных и итоговой согласованности (eventual consistency).

Ключевые принципы архитектуры:

- **Клиентское шифрование** — все данные шифруются на стороне клиента до передачи в сеть. Ни один узел, включая платформу, не имеет доступа к содержимому файлов.
- **Офлайн-первый** (offline-first) — все операции возможны без постоянного подключения к сети; синхронизация происходит при восстановлении связи.
- **Экономическая устойчивость** — автоматизированная биллинговая система с заморозкой, тарифами и распределением доходов между хост-провайдерами.
- **Песочница для хостов** — каждый узел хранения выполняется в изолированной среде с ограничениями seccomp и cgroups.

Система ориентирована на юрисдикцию Российской Федерации и соответствует требованиям 152-ФЗ (персональные данные) и 54-ФЗ (кассовые аппараты).

## Поток данных: Загрузка файла

```
Файл пользователя
       │
       ▼
  ┌─────────────┐
  │  Чанкинг     │  Разбиение файла на блоки по 4 МиБ
  │  (4 МиБ)     │  Последний блок добивается до полного размера
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │  Шифрование  │  XChaCha20-Poly1305
  │              │  Ключ деривируется из мастер-пароля через Argon2id
  │              │  Уникальный 24-байт nonce на каждый чанк
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │ Erasure      │  Reed-Solomon 3/5 (3 данных + 2 чётности)
  │ Coding       │  Каждый зашифрованный чанк → 5 шардов
  │              │  Достаточно 3 из 5 для восстановления
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │ Распределение│  Шарды распределяются по узлам сети через Kademlia DHT
  │ по узлам     │  Формируется Merkle-дерево для проверки целостности
  │              │  Метаданные записываются в ledger
  └─────────────┘
```

### Детали загрузки

1. **Чанкинг**: Файл разбивается на блоки фиксированного размера 4 МиБ (4 194 304 байт). Последний блок дополняется случайными байтами (padding), чтобы его размер был кратен размеру чанка. Padding записывается в метаданные и отсекается при восстановлении.

2. **Шифрование**: Каждый чанк шифруется поточным шифром XChaCha20-Poly1305 с AEAD. Ключ шифрования выводится из мастер-пароля пользователя с помощью функции вывода ключа Argon2id (параметры: m=65536, t=3, p=4). Nonce длиной 24 байта генерируется cryptographically secure RNG и сохраняется в метаданных чанка. Гарантируется уникальность nonce — при совпадении генерируется новый.

3. **Erasure Coding**: Каждый зашифрованный чанк кодируется кодом Рида-Соломона с параметрами (3, 5): из 3 фрагментов данных и 2 чётностных получается 5 шардов. Для восстановления исходных данных достаточно любых 3 шардов из 5, что позволяет потерять до 40% шардов без потери данных.

4. **Распределение**: Шарды отправляются на узлы сети, отобранные через Kademlia DHT. Каждый узел подтверждает приём. Merkle-дерево строится по хешам всех шардов; корневой хеш записывается в распределённый реестр (ledger) и gossip-протоколом распространяется по сети.

## Поток данных: Скачивание файла

```
Запрос на скачивание
       │
       ▼
  ┌──────────────────┐
  │ Запрос шардов     │  Ledger предоставляет список узлов, хранящих шарды
  │ у узлов           │  Запрашивается как минимум 3 из 5 шардов через P2P
  └────────┬─────────┘
           │
           ▼
  ┌──────────────────┐
  │ Сборка            │  Reed-Solomon декодирование: 3+ шардов → исходный чанк
  │ (Reassembly)      │  Проверка Merkle-хеша каждого восстановленного чанка
  └────────┬─────────┘
           │
           ▼
  ┌──────────────────┐
  │ Расшифрование     │  XChaCha20-Poly1305 AEAD verification
  │                  │  Проверка аутентичности тега + дешифрование
  │                  │  Удаление padding-байтов из последнего чанка
  └────────┬─────────┘
           │
           ▼
  ┌──────────────────┐
  │ Восстановление   │  Склейка чанков в исходном порядке
  │ файла            │  Проверка общего хеша (SHA-256)
  └──────────────────┘
```

### Детали скачивания

1. **Запрос шардов**: Клиент обращается к ledger за картой распределения шардов. Для каждого чанка запрашиваются шарды у доступных узлов. Если узел недоступен, запрашивается запасной шард. Система использует таймауты и exponential backoff при повторных попытках.

2. **Сборка**: Полученные шард(ы) декодируются алгоритмом Рида-Соломона. Минимальное количество — 3 из 5. Merkle-хеш каждого восстановленного чанка сравнивается с эталонным из ledger. При несовпадении — повторный запрос запасных шардов.

3. **Расшифрование**: AEAD-тег проверяется автоматически при расшифровании XChaCha20-Poly1305. При несовпадении тега генерируется ошибка целостности. Padding отсекается согласно метаданным чанка.

## Философия офлайн-первого (Offline-First)

VaultKeeper P2P спроектирован по принципу «офлайн-первый»: все критические операции должны выполняться локально, без постоянного подключения к сети.

### Локальные операции (без сети)

- Шифрование и расшифрование файлов
- Чанкинг и erasure coding
- Биллинговые расчёты и ведение баланса
- Запись в локальный ledger (SQLite)
- Генерация и восстановление ключей BIP39

### Сетевые операции (при наличии связи)

- Обмен gossip-сообщениями (синхронизация ledger)
- Отправка/получение шардов
- Heartbeat-проверки (каждые 15 минут)
- Proof-of-Storage вызовы
- Запросы на вывод средств

### Стратегия синхронизации

При восстановлении связи узел:

1. Отправляет свои локальные изменения через gossip
2. Запрашивает пропущенные обновления у соседей
3. Разрешает конфликты по правилу «последняя запись побеждает» (LWW) с Merkle-корнем как арбитром
4. Обновляет баланс и статус подписок

## Модель консенсуса

### Итоговая согласованность (Eventual Consistency)

Система не использует классические алгоритмы консенсуса (RAFT, PBFT). Вместо этого применяется модель итоговой согласованности, основанная на:

- **Merkle-реестре**: Каждое изменение записывается в локальный SQLite с Merkle-корнем. Корень служит компактной свёрткой (fingerprint) состояния.
- **Gossip-протокол**: Узлы периодически обмениваются Merkle-корнями. При обнаружении расхождения выполняется синхронизация на уровне отдельных записей.
- **LWW-разрешение конфликтов**: При конфликте двух версий одной записи побеждает версия с бóльшим логическим времением (timestamp). Если временные метки совпадают — побеждает запись с меньшим хешем (tiebreaker).

### Преимущества модели

- **Масштабируемость**: O(log N) сложность синхронизации через Merkle-деревья
- **Отказоустойчивость**: Переживает разделение сети (network partitions)
- **Простота**: Не требует кворума, выборов лидера или сложных протоколов

### Ограничения

- Возможны временные расхождения в балансе между узлами (сглаживаются в течение минут)
- Конфликты при одновременной модификации одной записи разрешаются детерминистически

## Модель безопасности

### Клиентское шифрование

Все данные шифруются на устройстве пользователя до передачи в сеть. Ключ шифрования никогда не покидает устройство. Платформа и узлы-хосты работают только с зашифрованными шардами.

### Изоляция хостов (песочница)

Каждый узел-хост выполняется в изолированной среде:

- **seccomp-bpf**: Белый список системных вызовов. Хост может выполнять только чтение/запись файлов, mmap, futex, epoll. Системные вызовы execve, clone, mount, ptrace заблокированы.
- **cgroups v2**: Ограничения на потребление CPU (максимум 50% одного ядра), памяти (512 МиБ), дисковых операций (100 IOPS), количество процессов (max 32).

### Аутентификация

- Узлы аутентифицируются по ed25519-ключам
- Каждый узел имеет уникальный Peer ID в libp2p
- M TLS для API-соединений (клиент ↔ API-сервер)

### Примечания

- На Windows хост-функционал (песочница seccomp/cgroups) недоступен. Пользователь может загружать и скачивать файлы, но не принимать чужие шард(ы).
- На macOS хост-функционал экспериментальный: sandbox-apparmor заменяется на sandbox-exec, ограничения менее строгие.
- Для продакшн-хостов рекомендуется Ubuntu 22.04 LTS или Debian 12.

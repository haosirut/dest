# Модель безопасности VaultKeeper P2P

## Модель угроз

### Что защищает система

VaultKeeper P2P защищает пользовательские данные от следующих угроз:

1. **Злоумышленные узлы-хосты**: Узлы сети получают зашифрованные шард(ы) и не могут прочитать содержимое, определить имя файла или тип данных.
2. **Перехват трафика (MITM)**: Все коммуникации между узлами защищены транспортным шифрованием libp2p (TLS 1.3 / Noise Protocol).
3. **Компрометация платформы**: Даже при полном доступе к серверам платформы невозможно расшифровать данные пользователей — ключи хранятся только у клиентов.
4. **Удаление данных хостом**: Erasure coding (3/5) гарантирует восстановление при потере до 2 шардов из 5 на каждый чанк.
5. **Подмена данных**: Merkle-деревья и AEAD-теги обеспечивают обнаружение любой модификации шардов.
6. **Атаки на хост-узел (RCE)**: Песочница seccomp + cgroups ограничивает возможный ущерб при компрометации процесса хоста.

### Что система НЕ защищает от

- **Утечка мастер-пароля**: Если пароль скомпрометирован, все данные могут быть расшифрованы.
- **DoS-атаки на сеть**: Отказ в обслуживании отдельных узлов возможен, но не влияет на целостность данных.
- **Физический доступ к устройству пользователя**: При получении доступа к устройству и незаблокированной сессии ключи могут быть извлечены из памяти.
- **Анализ трафика (traffic analysis)**: Определение факта обмена данными возможно, хотя содержимое скрыто.

## Шифрование

### Алгоритм: XChaCha20-Poly1305

VaultKeeper P2P использует потоковый шифр XChaCha20-Poly1305 с аутентифицированным дополнительным данными (AEAD) для шифрования каждого чанка.

**Параметры:**
- Ключ: 256 бит (32 байта)
- Nonce: 192 бит (24 байта)
- Тег аутентичности: 128 бит (16 байт)
- Дополнительные данные (AAD): хеш чанка (SHA-256) для привязки шифрования к конкретному блоку

**Почему XChaCha20-Poly1305:**
- Расширенный nonce (24 байта) практически исключает коллизии nonce при случайной генерации
- AEAD обеспечивает конфиденциальность и целостность в одной операции
- Аппаратное ускорение на современных процессорах (SIMD-инструкции)
- Рекомендован IETF (RFC 8439) и используется в TLS 1.3

### Вывод ключа: Argon2id

Мастер-пароль пользователя преобразуется в ключ шифрования с помощью функции вывода ключа Argon2id.

**Параметры:**
- Память: 64 МиБ (m=65536 KiB)
- Итерации: 3
- Параллелизм: 4 потока
- Длина выходного ключа: 32 байта
- Соль: 16 байт, генерируется случайно при инициализации хранилища

Argon2id выбран как победитель конкурса Password Hashing Competition (PHC). Комбинация режима с зависимостью от пароля и независимой от него части обеспечивает устойчивость как к side-channel атакам, так и к GPU-атакам.

### Уникальность Nonce

Nonce длиной 24 байта генерируется с помощью криптографически стойкого генератора псевдослучайных чисел (CSPRNG) — `ring::rand::SystemRandom`. При обнаружении коллизии nonce (практически невозможно с 192 битами) генерируется новый nonce. Nonce хранится в открытом виде в метаданных чанка — это безопасно, так как безопасность XChaCha20 не зависит от секретности nonce.

## Управление ключами

### BIP39 Mnemonic Recovery

При инициализации хранилища генерируется 24-словная мнемоническая фраза по стандарту BIP39. Эта фраза позволяет восстановить ключ шифрования и доступ ко всем данным при утере устройства.

**Процесс:**
1. CSPRNG генерирует 256 бит энтропии
2. Энтропия кодируется в 24 слова из словаря BIP39 (английский)
3. Из мнемоники выводится seed (BIP39 seed) через PBKDF2-HMAC-SHA512
4. Из seed выводится мастер-ключ шифрования через HKDF-SHA256
5. Мастер-пароль пользователя комбинируется с мастер-ключом через Argon2id для получения финального ключа шифрования

**Безопасность мнемоники:**
- Энтропия 256 бит — перебор невозможен даже с квантовым компьютером
- Проверочная контрольная сумма (last word) обнаруживает ошибки записи
- Мнемоника отображается пользователю один раз и нигде не сохраняется автоматически

### Зануление ключей (Zeroization)

Все криптографические ключи в памяти обнуляются при удалении (drop). Используется реализация `Zeroize` trait из крейта `zeroize`:

```rust
impl Drop for EncryptionKey {
    fn drop(&mut self) {
        self.key_material.zeroize();
        self.nonce.zeroize();
    }
}
```

Скомпилированный код проходит проверку на отсутствие оптимизаций, которые могут убрать обнуление (атрибут `volatile` на `memset_s`-эквиваленте).

## Изоляция хостов

### seccomp-bpf

Каждый хост-процесс загружает фильтр seccomp-bpf при запуске. Фильтр ограничивает набор допустимых системных вызовов.

**Белый список разрешённых системных вызовов:**
- `read`, `write`, `pread64`, `pwrite64` — чтение/запись шардов
- `openat`, `close`, `fstat` — работа с файловыми дескрипторами
- `mmap`, `munmap`, `mprotect` — управление памятью
- `futex` — синхронизация
- `epoll_wait`, `epoll_ctl`, `eventfd2` — сетевое IO
- `sendto`, `recvfrom`, `sendmmsg`, `recvmmsg` — передача данных по сети
- `getrandom` — генерация случайных чисел
- `clock_gettime`, `nanosleep` — таймеры
- `exit`, `exit_group` — завершение

**Чёрный список (заблокированные системные вызовы):**
- `execve`, `execveat` — запуск новых процессов
- `clone`, `fork`, `vfork` — создание процессов/потоков (кроме начальных)
- `mount`, `umount2` — работа с файловыми системами
- `ptrace` — отладка других процессов
- `kill` — отправка сигналов другим процессам
- `socket` с доменом `AF_UNIX` — предотвращение IPC
- `chmod`, `chown`, `chroot` — изменение прав

### cgroups v2

Ресурсные ограничения через cgroups v2:

| Ресурс | Лимит | Назначение |
|--------|-------|------------|
| CPU | max 50% одного ядра | Предотвращение перегрузки |
| Память | 512 МиБ | Защита от OOM на хост-машине |
| IOPS | 100 операций/сек | Ограничение дисковой нагрузки |
| Количество процессов | max 32 | Защита от fork-bomb |
| PIDs | max 64 | Дополнительная защита от fork |

## Proof-of-Storage (Доказательство хранения)

### Merkle-дерево вызовов

Для проверки того, что хост действительно хранит назначенные шард(ы), используется протокол Proof-of-Storage на основе Merkle-деревьев.

**Механизм:**
1. Для каждого шарда строится Merkle-дерево по блокам фиксированного размера (4 КиБ)
2. Вызывающая сторона отправляет случайный индекс листа и ожидает путь от листа к корню (audit path)
3. Хост должен за ограниченное время (5 секунд) вернуть значение листа и audit path
4. Вызывающая сторона пересчитывает Merkle-корень и сравнивает с эталонным

**Параметры:**
- Частота вызовов: 1 раз в 15 минут для каждого шарда (совмещено с heartbeat)
- Случайность индекса: CSPRNG с seed, известным только вызывающей стороне
- Таймаут ответа: 5 секунд
- Количество неудачных вызовов до пометки узла как ненадёжного: 3 подряд

## Безопасность биллинга

### Decimal-арифметика

Все финансовые расчёты выполняются с типом `rust_decimal::Decimal` с точностью до 28 значащих цифр. Это полностью исключает ошибки округления, характерные для типов с плавающей точкой (f32, f64).

### Защита от потерь при округлении

- Все тарифы хранятся как точные десятичные дроби (0.30 RUB, 1.0, 1.5, 2.0)
- Промежуточные расчёты не округляются до финального результата
- Итоговая сумма округляется до копеек (2 знака) только при списании/начислении
- Округление всегда в сторону выгоды пользователя

### Механизм заморозки

Баланс проверяется при каждом heartbeat (15 минут). При баланс ≤ 0:
1. Аккаунт переходит в состояние «заморожен»
2. Хост-узлы получают уведомление через gossip
3. Даётся 48 часов на пополнение баланса или экспорт данных
4. По истечении 48 часов данные удаляются безвозвратно

## Безопасность реестра (Ledger)

### Целостность Merkle-корня

Каждая запись в ledger включает в себя Merkle-корень. При gossip-синхронизации узлы обмениваются корнями и сверяют их. Несовпадение корня указывает на несогласованность данных.

### Разрешение конфликтов

При обнаружении конфликта двух версий одной записи:
1. Сравниваются временные метки (timestamp)
2. Побеждает запись с более поздней меткой (Last-Writer-Wins)
3. При равных метках — побеждает запись с меньшим хешем (tiebreaker)
4. Запись о конфликте сохраняется в журнал аудита
